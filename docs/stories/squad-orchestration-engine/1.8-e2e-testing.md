## Acceptance Criteria

1. **AC1 — Test Framework:** Test suite implemented using Vitest (consistent with existing test infrastructure) with integration/E2E test patterns. Tests run in Node.js environment (not browser — testing backend orchestration).

2. **AC2 — API Mocking:** External API calls (DALL-E, Whisper, any future providers) mocked using MSW (Mock Service Worker) or manual mock modules. Mock responses are realistic and cover success, error, and timeout scenarios.

3. **AC3 — Full Flow Test:** Complete squad-copy pipeline test:
   - Trigger run via `SquadOrchestrator.execute()` with mock playbook
   - Validate all phases execute in order
   - Validate step outputs accumulate in context
   - Validate events.jsonl has correct event sequence
   - Validate final state is `completed` with outputs
   - Test completes in <30 seconds

4. **AC4 — Pause/Resume Test:** Test that verifies:
   - Run starts and reaches a specific phase
   - Pause is triggered via StateManager.pause()
   - State is saved to state.yaml with correct checkpoint
   - Resume loads state and continues from paused step
   - Final state is `completed` (not `paused`)

5. **AC5 — Abort Test:** Test that verifies:
   - Run starts execution
   - Abort is triggered via StateManager.abort()
   - Run status changes to `aborted`
   - No further steps execute after abort
   - Events include `run.aborted` event

6. **AC6 — Error Handling Test:** Test that verifies:
   - Mock an external API failure in a step
   - Step fails and error is captured
   - Run status changes to `failed`
   - Error details available in state and events
   - Events include `run.failed` with error data

7. **AC7 — Condition Engine Test:** Test that verifies:
   - Pre-condition blocker prevents step execution when not met
   - Post-condition failure marks step as failed
   - Non-blocker conditions log warning but don't stop execution

8. **AC8 — Event Sourcing Test:** Test that verifies:
   - Full run generates correct event sequence in events.jsonl
   - EventStore.replay() reconstructs state matching actual final state
   - Corrupted state.yaml recovers via event replay

9. **AC9 — CI Integration:** Tests run as part of `npm test` command. All E2E tests complete in <60 seconds total. Test results are deterministic (no flaky tests due to timing).

## Dev Notes

### Risks

- **Flaky tests (timing):** Pause/resume tests depend on timing — use deterministic triggers (e.g., mock step that calls pause callback) instead of setTimeout. Mitigação: Task 9.5 executa 3x para detectar flakiness.
- **MSW + CommonJS:** MSW é ESM-first. Para módulos CommonJS do squad engine, preferir manual mocks (`vi.mock()`) ou mock factories em vez de MSW interceptors. MSW é melhor para API route tests (HTTP level).
- **CI timeout:** E2E tests com file system I/O podem ser mais lentos em CI. Mitigação: AC9 limita a <60s total, usar `os.tmpdir()` (SSD-backed).

### Previous Stories Insights

- **Testing Pattern:** Story 1.5 used Vitest with `vi.mock()` for CommonJS modules. Some mocks had limitations — prefer integration tests that use real modules with mock data. [Source: Story 1.5 TESTING_NOTES.md]
- **File System Pattern:** Use `os.tmpdir()` for test run directories to avoid polluting real `.aios/` directory. [Best practice]
- **CommonJS Pattern:** Squad engine modules use `require()`. Tests may need `vi.mock()` for specific dependencies. [Source: Story 1.5]
- **MSW Integration:** MSW works best for HTTP-level mocking (API route tests). For internal module mocking (squad engine components), use `vi.mock()` with manual factories. Do NOT use MSW for SquadOrchestrator/TaskExecutor internals. [Source: PO validation]
- **Timeout:** Set Vitest timeout to 30s for E2E tests: `test.setTimeout(30000)` or configure in vitest.config.ts. [Recommendation]
