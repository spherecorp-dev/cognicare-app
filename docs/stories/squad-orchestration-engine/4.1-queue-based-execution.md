## Story

**As a** AIOS framework,
**I want** queue-based execution para múltiplos runs simultâneos usando BullMQ,
**so that** posso escalar para 100+ runs sem bloquear o processo principal.

---

## Acceptance Criteria

1. **AC1 — BullMQ Integration:** BullMQ instalado e configurado com Redis como backend. Queue `squad-runs` criada com configuração de concurrency, retry, e backoff.

2. **AC2 — Enqueue Instead of Direct Execution:** `POST /api/runs` enfileira job na queue em vez de executar diretamente. Job contém: `{ squadId, trigger, overrides, priority }`. Retorna `{ runId, queued: true }` imediatamente (async).

3. **AC3 — Worker Processing:** Workers (max 10 concurrent por padrão, configurável) processam jobs da fila. Cada worker instancia `SquadOrchestrator.execute()` com os parâmetros do job. Worker lifecycle: receive job → execute → report completion.

4. **AC4 — Priority Support:** Runs aceitam campo `priority` (1-10, default: 5). Priority 1 = máxima urgência. Jobs com priority maior são processados primeiro. Dashboard mostra priority na lista de runs.

5. **AC5 — Dead Letter Queue (DLQ):** Failed jobs vão para DLQ (`squad-runs-dlq`) após 3 tentativas automáticas de BullMQ. DLQ entries incluem: job data, error stack, attempt count, timestamps. Dashboard mostra DLQ count e permite retry manual.

6. **AC6 — Dashboard Queue Status:** Dashboard mostra métricas de queue em tempo real:
   - Queue size (waiting + active + delayed)
   - Workers ativos / max workers
   - Jobs completed (24h)
   - DLQ count
   - Throughput (jobs/min)

7. **AC7 — Graceful Shutdown:** Workers fazem graceful shutdown: completam job atual antes de parar. `SIGTERM` signal handler implementado. Nenhum job é perdido ou duplicado durante restart.

8. **AC8 — Queue Events Logging:** Eventos de queue (job.added, job.active, job.completed, job.failed, job.stalled) logados em `.aios/queue/log.jsonl`. EventStore do run também registra `run.queued` e `run.dequeued` events.

## Dev Notes

### Previous Stories Insights

- **SquadOrchestrator.execute():** Aceita `{ squadId, trigger, overrides }` e retorna `{ runId, status }`. Queue wraps this call. [Source: Story 1.1, 3.3]
- **State Manager:** Cria state em `.aios/squad-runs/{runId}/state.yaml` durante execute(). State deve ser criado ANTES de enqueue para que dashboard mostre run imediatamente. [Source: Story 1.4]
- **EventStore:** `EventStore.append(runId, event)` para registrar queue events no run. [Source: Story 1.7]
- **Dashboard Polling:** Frontend usa `useQuery` com `refetchInterval: 2000`. Queue metrics podem usar intervalo maior (5s). [Source: Story 1.6]
- **Override Propagation:** Overrides passados no trigger são preservados no job data e propagados para SquadOrchestrator. [Source: Story 3.3]

### BullMQ Configuration

```javascript
// queue-config.js
const QUEUE_CONFIG = {
  connection: {
    url: process.env.REDIS_URL || 'redis://localhost:6379'
  },
  defaultJobOptions: {
    attempts: 3,
    backoff: {
      type: 'exponential',
      delay: 5000  // 5s, 10s, 20s
    },
    removeOnComplete: {
      age: 86400,  // 24h
      count: 1000
    },
    removeOnFail: false  // Keep in DLQ
  },
  workerOptions: {
    concurrency: parseInt(process.env.MAX_QUEUE_WORKERS || '10'),
    lockDuration: 1800000  // 30min (max pipeline duration)
  }
};
```

### Priority Mapping

```javascript
// BullMQ priority: lower number = higher priority
// Our API: 1 = highest, 10 = lowest
// Mapping: BullMQ priority = our priority (same direction)
const PRIORITY_MAP = {
  1: 1,   // Urgent
  5: 5,   // Normal (default)
  10: 10  // Low
};
```

### Risks

- **Redis dependency:** BullMQ requer Redis. Se Redis unavailable, queue não funciona. Mitigação: fallback para direct execution quando Redis está down (backward compat flag).
- **Job stalling:** Se worker crasha durante job processing, job fica "stalled". Mitigação: BullMQ auto-recovery (stalledInterval: 30s) re-enfileira stalled jobs.
- **Memory pressure:** 100+ runs simultâneos podem consumir muita RAM nos workers. Mitigação: Limitar concurrency via MAX_QUEUE_WORKERS, monitorar memory usage.
- **Lock duration:** Pipeline pode durar 30+ min. Se lock expira, BullMQ pode re-processar job. Mitigação: lockDuration alto (30min) + extend lock durante processing.
