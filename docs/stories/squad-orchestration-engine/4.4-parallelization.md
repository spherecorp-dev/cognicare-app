## Story

**As a** squad orchestrator,
**I want** executar steps independentes em paralelo dentro de um pipeline,
**so that** posso reduzir tempo total de pipeline (ex: gerar ad-copy para FR, ES, EN em paralelo: 9min → 3min).

---

## Acceptance Criteria

1. **AC1 — Parallel Group Type:** Playbook suporta novo tipo de step: `type: parallel_group`. Um parallel group contém array de tasks que executam simultaneamente. Validação de playbook aceita e valida parallel groups.

2. **AC2 — Parallel Execution:** `ParallelExecutor.execute()` executa todas as tasks do grupo via `Promise.allSettled()`. Cada task roda independentemente com seu próprio context snapshot. Resultados são agregados em um objeto `{ [taskId]: result }`.

3. **AC3 — Wait for All:** `wait_for_all: true` (default) aguarda todas as tasks completarem antes de prosseguir. O parallel group só completa quando todas as tasks completam (ou falham, conforme `fail_fast` config).

4. **AC4 — Fail Fast:** Se `fail_fast: true`, quando uma task falha, cancela as demais tasks pendentes (via AbortController). Se `fail_fast: false` (default), todas as tasks completam independentemente de falhas individuais. Resultados incluem status de cada task (completed/failed).

5. **AC5 — Context Merge:** Outputs de todas as tasks do parallel group são merged no context acumulado. Merge strategy: cada task contribui para `context.parallel_results[groupId]`. Nenhuma task no grupo pode sobrescrever output de outra (isolamento).

6. **AC6 — Event Emission:** Eventos emitidos para cada task individual dentro do grupo:
   - `parallel_group.started` com lista de tasks
   - `step.started` / `step.completed` / `step.failed` para cada task
   - `parallel_group.completed` com summary (total, succeeded, failed, duration)

7. **AC7 — Dashboard Visualization:** Dashboard mostra parallel groups como bloco visual expandível:
   - Grupo aparece como single node no pipeline com badge "Parallel (3 tasks)"
   - Expandir mostra tasks individuais com status
   - Progress: "2/3 completed" com progress bar
   - Duration: mostra tempo do grupo (wall clock, não sum of tasks)

8. **AC8 — Playbook Schema:** Pipeline schema atualizado para aceitar parallel groups:
   ```yaml
   steps:
     - id: generate-multilang
       type: parallel_group
       wait_for_all: true
       fail_fast: false
       tasks:
         - id: gen-fr
           task: generate-ad-copy
           input: { language: "fr" }
         - id: gen-es
           task: generate-ad-copy
           input: { language: "es" }
         - id: gen-en
           task: generate-ad-copy
           input: { language: "en" }
       on_success: next_step
   ```

## Dev Notes

### Previous Stories Insights

- **Pipeline Execution Loop (Story 1.1):** `SquadOrchestrator` iterates steps sequentially. Parallel group introduces branching in this loop. [Source: Story 1.1]
- **TaskExecutor (Story 1.2):** `executeTask(step, context)` is the atomic unit. ParallelExecutor calls this N times. [Source: Story 1.2]
- **Condition Engine (Story 1.3):** Pre/post-conditions work at step level. ParallelExecutor must validate conditions for each task independently. [Source: Story 1.3]
- **Event Sourcing (Story 1.7):** EventStore is append-only. Parallel tasks may emit events concurrently — EventStore must handle concurrent appends. [Source: Story 1.7]
- **Skip Phases (Story 3.3):** If a parallel group's phase is in `skip_phases`, entire group is skipped. [Source: Story 3.3]
- **Retry Logic (Story 2.4):** RetryHandler wraps individual task execution. Each task in parallel group has independent retry. [Source: Story 2.4]
- **PRD Reference:** parallel-executor.js mencionado no PRD como parte da arquitetura planejada. [Source: PRD 3.2]

### Parallel Execution Flow

```
Sequential Pipeline:
  step-1 → step-2 → [parallel_group] → step-4
                         ↓
                    ┌─ task-a ─┐
                    ├─ task-b ─┤  (concurrent)
                    └─ task-c ─┘
                         ↓
                    merge results
                         ↓
                      step-4

Timeline:
  |--step-1--|--step-2--|--parallel--|--step-4--|
                        |  task-a   |
                        |  task-b   |
                        |  task-c   |
                        (wall clock = max(a,b,c))
```

### Context Isolation Strategy

```javascript
// Each parallel task gets a deep clone of context
// This prevents race conditions and data leaks between tasks
const cloneContext = (ctx) => JSON.parse(JSON.stringify(ctx));

// After all tasks complete, results are merged:
context.parallel_results = {
  'generate-multilang': {
    'gen-fr': { status: 'completed', output: { copy: '...' } },
    'gen-es': { status: 'completed', output: { copy: '...' } },
    'gen-en': { status: 'failed', error: 'Rate limit exceeded' }
  }
};

// Next step accesses via:
// {{parallel_results.generate-multilang.gen-fr.output.copy}}
```

### Risks

- **Race conditions:** Parallel tasks share same EventStore (append-only). Concurrent appends must be atomic. Mitigação: EventStore uses file locks (or Redis SET for atomicity).
- **Memory pressure:** Deep-cloning context for each parallel task multiplies memory usage. Mitigação: Limit parallel group to max 10 tasks. Context snapshots are typically <100KB each.
- **Error cascading:** In fail_fast mode, AbortController must cleanly cancel in-flight tasks. Mitigação: Tasks must check AbortSignal and exit gracefully.
- **Pipeline validation complexity:** Parallel groups add complexity to schema validation. Mitigação: Explicit rejection of nested parallel groups. Clear error messages.
- **Context merge conflicts:** If two parallel tasks produce output with same key. Mitigação: Each task writes to its own namespace `parallel_results[groupId][taskId]`.

### Concerns (non-blocking)

Functionally harmless (EventStore.append doesn't validate types) but canonical list is incomplete.

2. **MEDIUM — ConditionEngine not wired in orchestrator path:** `executeParallelGroup()` (squad-orchestrator.js:321) creates ParallelExecutor without passing conditionEngine. Pre/post-conditions for parallel tasks won't be validated via orchestrator path. ParallelExecutor supports it — wiring is missing.

3. **LOW — STATUS_ICONS inconsistency:** ParallelGroupNode uses `Circle` for `paused` status while StepDetails uses `PauseCircle`. Duplicated STATUS_ICONS map and formatDuration() between components.
