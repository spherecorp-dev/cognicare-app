## Story

**As a** Squad Orchestration Engine,
**I want** circuit breaker para APIs externas (DALL-E, Whisper, Meta, TikTok),
**so that** não desperdice retries quando uma API está down e proteja o sistema de cascading failures.

---

## Acceptance Criteria

1. **AC1 — Circuit Breaker Class:** `APICircuitBreaker` implementado em `.aios-core/core/orchestration/squad-engine/circuit-breaker.js` com 3 estados: `CLOSED` (normal), `OPEN` (bloqueando), `HALF_OPEN` (testando).

2. **AC2 — State Transitions:**
   - `CLOSED` → `OPEN`: Após 5 falhas consecutivas (configurable `failureThreshold`)
   - `OPEN` → `HALF_OPEN`: Após timeout de 60s (configurable `resetTimeout`)
   - `HALF_OPEN` → `CLOSED`: Se próximo call for sucesso
   - `HALF_OPEN` → `OPEN`: Se próximo call falhar

3. **AC3 — Circuit Breaker Error:** `CircuitBreakerOpenError` lançado quando circuito está `OPEN` — não tenta chamar a API, fail-fast.

4. **AC4 — Per-Provider Breakers:** Cada provider de API externa tem seu próprio circuit breaker independente (ex: DALL-E breaker separado de Whisper breaker). Registry: `CircuitBreakerRegistry` mantém breakers por nome.

5. **AC5 — Event Logging:** Transições de estado emitidas via EventStore:
   - `circuit.opened` — data: `{ provider, failures, threshold }`
   - `circuit.half_opened` — data: `{ provider, timeout_ms }`
   - `circuit.closed` — data: `{ provider, recovered: true }`

6. **AC6 — Call Wrapper:** `circuitBreaker.call(fn)` wraps qualquer async function com circuit breaker logic. Retorna resultado se sucesso, registra falha se erro.

7. **AC7 — Status API:** `CircuitBreakerRegistry.getStatus()` retorna mapa de todos os breakers com estado atual, último erro, e contadores. Disponível via `GET /api/system/circuit-breakers`.

8. **AC8 — Dashboard Visibility:** Endpoint `/api/system/circuit-breakers` retorna status de todos os breakers. Dashboard pode consultar para exibir status (UI é out of scope).

## Dev Notes

### Previous Stories Insights

- **CommonJS Pattern:** Squad engine uses `require()`. Follow same for circuit-breaker.js. [Source: Story 1.5]
- **Error Class Pattern:** Story 1.4 defines StateCorruptionError, InvalidStateError. Follow same pattern for CircuitBreakerOpenError. [Source: Story 1.4]
- **API Route Pattern:** Next.js 16 route handlers use `params: Promise<>`. Status endpoint is simple GET without params. [Source: Story 1.5]
- **EventStore API:** `eventStore.append(runId, eventType, data)` — for circuit events, runId may be optional (system-level events). Consider using a system runId or null. [Source: Story 1.7]
- **NFR2:** PRD requires circuit breaker after 3 consecutive failures with 2min timeout. Story AC uses 5 failures / 60s as configurable defaults (more granular than PRD minimum). [Source: PRD NFR2]

### State Machine Diagram

```
     success
  ┌────────────┐
  │            ▼
CLOSED ───[5 failures]───► OPEN ───[60s timeout]───► HALF_OPEN
  ▲                                                      │
  │           success                                    │
  └──────────────────────────────────────────────────────┘
                          failure
                    HALF_OPEN ──────────► OPEN
```

### PRD Requirements Reference

- **NFR2:** Circuit Breaker para APIs externas — após 3-5 falhas consecutivas, abrir circuito
- **PRD Section 5.11:** Detailed acceptance criteria for Circuit Breaker
- **PRD Section 3.6.5:** Error handling per integration (DALL-E, Whisper, Meta, TikTok)

### Risks

- **Timer-based tests:** HALF_OPEN transition depends on setTimeout. Use jest.useFakeTimers() to make tests deterministic. [Best practice]
- **Memory leak:** CircuitBreakerRegistry is singleton — breakers persist for server lifetime. Mitigação: Registry.reset() for cleanup. MVP acceptable.
- **System-level events:** Circuit breaker events don't have a runId context. May need to pass current runId through call chain or use a system event channel.

### Test Scenarios

| Scenario | Expected |
|----------|----------|
| 4 consecutive failures | State remains CLOSED, counter = 4 |
| 5 consecutive failures | State transitions to OPEN |
| Call while OPEN | Throws CircuitBreakerOpenError immediately |
| 60s passes while OPEN | State transitions to HALF_OPEN |
| Success in HALF_OPEN | State transitions to CLOSED, counter reset |
| Failure in HALF_OPEN | State transitions back to OPEN |
| Success resets counter | After 3 failures + 1 success, counter = 0 |
| Per-provider isolation | Provider A OPEN, Provider B CLOSED |
