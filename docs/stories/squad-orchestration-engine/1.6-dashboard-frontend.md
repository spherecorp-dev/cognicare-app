## Story

**As a** squad operator monitoring pipeline executions,
**I want** a real-time dashboard UI showing pipeline progress with phase visualization, step details, and execution controls,
**so that** I can monitor squad-copy runs, understand current execution state, and intervene (pause/resume/abort) when needed.

---

## Acceptance Criteria

1. **AC1 — Runs List Page:** A `/runs` page displays all squad executions in a card grid or table with columns: RunID, SquadID, Status (color-coded badge), Started At, Duration. Supports pagination (limit/offset) and status filter dropdown.

2. **AC2 — Run Detail Page:** Clicking a run navigates to `/runs/{runId}` showing a two-section layout: PipelineVisualizer (phase flow diagram) and StepDetails (current step information with input/output data).

3. **AC3 — PipelineVisualizer:** Displays pipeline phases as a vertical/horizontal flow diagram with color-coded nodes: Green (completed), Yellow/Amber (running), Gray (pending), Red (failed). Shows phase name, step count, and clickable phases for detail expansion.

4. **AC4 — Real-time Polling:** The detail page polls `GET /api/runs/{runId}/state` every 2 seconds using React Query's `refetchInterval`. Polling stops automatically when status is `completed`, `failed`, or `aborted`. UI updates reflect new currentStepIndex and phase progress without full page reload.

5. **AC5 — Controls (Pause/Resume/Abort):** A Controls component provides:
   - Pause button (enabled when status === 'running')
   - Resume button (enabled when status === 'paused')
   - Abort button (always enabled, with confirmation Dialog before executing)
   - All actions call `POST /api/runs/{runId}/control` with appropriate action
   - Show loading state during API call, success/error feedback after

6. **AC6 — Context & Outputs Display:** A ContextSnapshot component shows accumulated execution context per step (collapsible sections). When run completes, an OutputsDisplay section shows generated artifacts from `GET /api/runs/{runId}/outputs` with file name, size, and download links.

7. **AC7 — Error Handling:** Handle API errors gracefully:
   - 404 (Run not found): Show "Run not found" page with link to runs list
   - 409 (Invalid state transition): Show toast error with explanation
   - 500 (Server error): Show error banner with retry button
   - Network timeout: React Query auto-retries with backoff

8. **AC8 — Responsive Design:** Layout adapts to mobile screens: pipeline stacks vertically, controls remain accessible, text truncates appropriately. Follows Tailwind responsive patterns (`sm:`, `md:`, `lg:` breakpoints).

9. **AC9 — Accessibility:** Semantic HTML (article, section, nav), ARIA labels on interactive elements, keyboard navigation for controls, status communicated via text + icons (not color alone).

## Dev Notes

### Previous Story (1.5) Insights

- **Path Alias:** `@aios/*` maps to `../.aios-core/*` in tsconfig.json. Vitest uses `@aios` alias in vitest.config.ts. [Source: Story 1.5 TD-1.5.4 resolution]
- **Next.js 16 Params:** Route handlers use `params: Promise<{ runId: string }>` (NOT sync). Must `await params` before accessing. [Source: Story 1.5 Next.js 16 compatibility fix]
- **CommonJS Modules:** Squad engine uses `require()` with `// eslint-disable-next-line @typescript-eslint/no-require-imports`. [Source: Story 1.5 import pattern]
- **Test Limitation:** CommonJS mocking in Vitest has limitations (24/48 tests pending in Story 1.5). For frontend tests, use standard ES module mocking. [Source: Story 1.5 TESTING_NOTES.md]
- **Error Types:** All catch blocks use `catch (error: unknown)` with `(error as Error).message`. [Source: Story 1.5 TypeScript strict mode]
