## Story

**As a** Squad Orchestration Engine,
**I want** retry automático para erros transientes e compensating transactions para rollback,
**so that** posso recuperar de falhas sem intervenção manual.

---

## Acceptance Criteria

1. **AC1 — Retry with Exponential Backoff:** `RetryHandler` class em `.aios-core/core/orchestration/squad-engine/retry-handler.js` implementa retry com exponential backoff:
   - Rate limit (HTTP 429) → retry 3x: 2s, 4s, 8s delays
   - Timeout errors → retry 2x: 5s, 10s delays
   - Configurable via retry config no step definition

2. **AC2 — Non-Transient Error Detection:** Erros não-transientes (HTTP 400, 401, 403) → fail immediately sem retry. `RetryHandler` classifica erros como transient ou non-transient baseado em HTTP status code ou error type.

3. **AC3 — Compensating Transactions:** Playbook define `on_failure.compensate` em steps/phases:
   ```yaml
   on_failure:
     compensate: cleanup-downloaded-videos
   ```
   Quando uma fase falha, compensating transactions são executadas em ordem reversa (LIFO) para cleanup de recursos.

4. **AC4 — Compensating Transaction Executor:** `CompensationExecutor` class executa tasks de compensação via TaskExecutor. Compensations são best-effort (falha de compensação é logada mas não bloqueia).

5. **AC5 — Structured Retry Logging:** Cada retry attempt emitido via EventStore:
   - `step.retry` — data: `{ stepId, attempt, maxAttempts, delay_ms, error_type, error_message }`
   - `step.retry_exhausted` — data: `{ stepId, totalAttempts, final_error }`
   - `phase.compensating` — data: `{ phaseName, compensations: [...] }`

6. **AC6 — Integration with Circuit Breaker:** RetryHandler integra com APICircuitBreaker (Story 2.3):
   - `CircuitBreakerOpenError` → fail immediately (não retry, circuito aberto)
   - Retry failures count towards circuit breaker failure counter

7. **AC7 — Dashboard Timeline:** Retry attempts e compensations visíveis na timeline do dashboard via eventos (usando endpoint existente `/api/runs/{runId}/events`).

8. **AC8 — Retry Configuration:** Step definition suporta custom retry config:
   ```yaml
   retry:
     max_attempts: 3
     base_delay_ms: 2000
     strategy: exponential  # exponential | linear | fixed
     transient_errors: [429, 502, 503, 504, ETIMEDOUT, ECONNRESET]
   ```

## Dev Notes

### Previous Stories Insights

- **TaskExecutor Pattern:** TaskExecutor.executeTask() currently runs task directly. Wrap with RetryHandler.retryWithBackoff() for retry logic. [Source: Story 1.2]
- **CircuitBreaker Integration:** Story 2.3 provides `circuitBreaker.call(fn)` wrapper. RetryHandler wraps the circuit breaker call with retry. Order: RetryHandler → CircuitBreaker → actual API call. [Source: Story 2.3]
- **Fire-and-forget Events:** All event emissions use try/catch, non-blocking. [Source: Story 1.7]
- **IdempotencyCache:** Story 2.2 cache prevents actual re-execution on retry. RetryHandler + IdempotencyCache together: retry triggers cache check first. [Source: Story 2.2]

### Execution Flow with Retry + Circuit Breaker

```
TaskExecutor.executeTask(step)
  │
  ├─ IdempotencyCache.get(key) → cache hit? → return cached
  │
  ├─ RetryHandler.retryWithBackoff(
  │    attempt => CircuitBreaker.call(
  │      () => actualTask(input)
  │    )
  │  )
  │    ├─ Attempt 1: CircuitBreaker → API call → 429 error
  │    │    → isTransient(429) = true → wait 2s
  │    ├─ Attempt 2: CircuitBreaker → API call → 429 error
  │    │    → isTransient(429) = true → wait 4s
  │    ├─ Attempt 3: CircuitBreaker → API call → success
  │    │    → return result
  │    └─ If all fail: throw RetryExhaustedError
  │
  └─ IdempotencyCache.set(key, result)
```

### Compensating Transactions LIFO Order

```
Phase: production
  Step 1: download-spy-videos    ← compensate: cleanup-downloaded-videos
  Step 2: transcribe-videos      ← compensate: cleanup-transcriptions
  Step 3: generate-images-api    ← FAILS HERE

Compensation execution order (LIFO):
  1. cleanup-transcriptions (Step 2's compensation)
  2. cleanup-downloaded-videos (Step 1's compensation)
```

### Error Classification Table (from PRD)

| Error Type | HTTP Status | Transient? | Strategy |
|------------|-------------|------------|----------|
| Rate limit | 429 | Yes | 3x exponential: 2s, 4s, 8s |
| Server error | 502, 503, 504 | Yes | 3x exponential: 2s, 4s, 8s |
| Timeout | ETIMEDOUT | Yes | 2x: 5s, 10s |
| Connection reset | ECONNRESET | Yes | 2x: 5s, 10s |
| Bad request | 400 | No | Fail immediately |
| Unauthorized | 401 | No | Fail immediately |
| Forbidden | 403 | No | Fail immediately |
| Circuit open | CircuitBreakerOpenError | No | Fail immediately |

### Risks

- **Delay timing in tests:** Exponential backoff delays slow down tests. Mitigação: Use jest.useFakeTimers() or allow delay override in test mode.
- **Compensation task resolution:** Compensations reference task names. TaskExecutor must resolve them. Mitigação: Reuse existing TaskExecutor.loadTaskFile() pattern.
- **Cascading compensations:** If compensation itself needs retry, could create complexity. Mitigação: Compensations are best-effort, no retry.

**Observations (non-blocking):**
- `isTransientError()` supports both `error.statusCode` (Axios) and `error.status` (fetch API) — good compatibility
- Compensation step IDs use `compensate-{stepId}` prefix for traceability in event logs
- `retry: false` explicitly disables retry per step — clean opt-out mechanism
