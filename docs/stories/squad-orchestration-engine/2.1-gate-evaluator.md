## Story

**As a** Squad Orchestration Engine,
**I want** implementar gates de revisão com loops (max rounds),
**so that** posso executar cycles de review-fix até aprovação ou max rounds.

---

## Acceptance Criteria

1. **AC1 — Gate Evaluator Class:** `GateEvaluator` implementado em `.aios-core/core/orchestration/squad-engine/gate-evaluator.js` com método `evaluateReviewGate(gateConfig, context, eventStore)`.

2. **AC2 — Verdict Processing:** Processa 3 tipos de verdict:
   - `APPROVED` → continuar para `on_verdict.APPROVED` (next step/phase)
   - `REVISION_NEEDED` → loop para `on_verdict.REVISION_NEEDED` se `rounds < max_rounds`
   - `REJECTED` → ir para `on_verdict.REJECTED` ou descartar artefato

3. **AC3 — Review Rounds Counter:** Incrementa `context.review_rounds` a cada loop de revisão. Counter inicializa em 0 e persiste entre iterations.

4. **AC4 — Escalation on Max Rounds:** Se `rounds >= max_rounds`, emite evento `gate.escalated` via EventStore e bloqueia execução com `GateEscalationError`. Default `max_rounds: 3` (configurável no playbook).

5. **AC5 — Event Logging:** Todos os verdicts registrados via EventStore:
   - `gate.evaluated` — data: `{ gateId, verdict, round, max_rounds, reviewer }`
   - `gate.escalated` — data: `{ gateId, round, reason: "max_rounds_reached" }`

6. **AC6 — Playbook Gate Schema:** Gates definidos no playbook YAML com estrutura:
   ```yaml
   - id: review-image-concept
     type: review_gate
     reviewer: "@copy-chief"
     max_rounds: 2
     on_verdict:
       APPROVED: generate_image_prompts
       REVISION_NEEDED: image_revision_loop
       REJECTED: discard_concept
   ```

7. **AC7 — SquadOrchestrator Integration:** `SquadOrchestrator` detecta steps do tipo `review_gate` e delega para `GateEvaluator` ao invés de `TaskExecutor`. GateEvaluator é instanciado com acesso ao EventStore.

8. **AC8 — Dashboard Timeline:** Events de gate (`gate.evaluated`, `gate.escalated`) são visíveis na timeline do dashboard via endpoint `/api/runs/{runId}/events` (já existente da Story 1.7).

### Gate Config Schema (Playbook YAML)

```yaml
# Example step in pipeline.yaml
- id: review-image-concept
  type: review_gate
  reviewer: "@copy-chief"
  max_rounds: 2
  on_verdict:
    APPROVED: generate_image_prompts    # stepId to jump to
    REVISION_NEEDED: image_revision_loop # stepId to loop back to
    REJECTED: discard_concept           # stepId for rejection
```

### PRD Requirements Reference

- **FR4:** Gates de revisão com APPROVED/REVISION_NEEDED/REJECTED verdicts
- **NFR2:** Success rate 95%+ (gate loops contribute to reliability)
- **PRD Section 5.10:** Detailed acceptance criteria for Gate Evaluator

### Risks

- **Loop infinito:** Se max_rounds não é validado, gate pode looper indefinidamente. Mitigação: Hard cap de max_rounds (máximo 10, default 3).
- **Step navigation:** REVISION_NEEDED precisa navegar para step anterior no pipeline. SquadOrchestrator precisa suportar "jump to step" ao invés de apenas "next step". Mitigação: Implementar step lookup by ID.
