## Story

**As a** n8n workflow,
**I want** um webhook API production-ready que conecte ao SquadOrchestrator real e suporte múltiplos event types com segurança reforçada,
**so that** posso triggerar pipelines automaticamente quando ofertas são criadas, com garantias de segurança e rastreabilidade.

---

## Acceptance Criteria

1. **AC1 — Webhook-to-Orchestrator Connection:** O endpoint `POST /api/webhooks/n8n` (implementado em Story 1.9) conecta ao `SquadOrchestrator.execute()` real, substituindo o TODO placeholder (`route.ts:176`). Webhook trigger cria run real com `trigger.type = "webhook"`, `trigger.source = "n8n"`, e `trigger.callback_url` armazenado no state.

2. **AC2 — Multiple Event Types:** Além de `offer.created`, suporta `manual_trigger` (trigger manual via n8n) e `squad.retry` (re-executar run que falhou). Event handler extensível via registry pattern: `webhookEventHandlers.register(eventName, handler)`.

3. **AC3 — Webhook Signature Verification (HMAC):** Valida `X-Webhook-Signature` header usando HMAC-SHA256 com `N8N_WEBHOOK_SECRET`. Se signature inválida e `WEBHOOK_REQUIRE_SIGNATURE=true`, retorna `401`. Se env var não configurada, fallback para Bearer token auth (backward compatible com Story 1.9).

4. **AC4 — HTTPS-Only Callbacks:** Em produção (`NODE_ENV=production`), `callback_url` deve ser HTTPS. HTTP rejeitado com `400 { error: "HTTPS required for callbacks in production" }`. Em development, HTTP permitido. IPv6 SSRF ranges `fd00::/8` e `fe80::/10` também bloqueados.

5. **AC5 — Callback Lifecycle Integration:** Callback é executado automaticamente quando run completa, falha ou é abortado. Integra com EventStore (Story 1.7): listener em `run.completed`, `run.failed`, `run.aborted` dispara callback. Payload: `{ runId, status, squadId, duration_ms, outputs_path, error?, trigger }`.

6. **AC6 — Webhook Management Endpoints:** `GET /api/webhooks/history` retorna últimas 100 entries do webhook log com paginação (`?limit=N&offset=N`). `GET /api/webhooks/stats` retorna estatísticas agregadas: total requests, success/error counts, average response time, top events.

7. **AC7 — E2E Integration Test:** Test end-to-end: webhook request → SquadOrchestrator.execute() chamado → run criado → run completa (mock) → callback enviado → webhook log contém entry. Usa mocked SquadOrchestrator (não executa pipeline real).

- **Story 1.9 Foundation:** Webhook endpoint, Bearer auth, rate limiting (100 req/min), callback system (3x retry), JSONL logging, input validation, SSRF prevention (IPv4) — all already implemented. This story EXTENDS and HARDENS 1.9. [Source: Story 1.9]

### Risks

- **ESM/CJS Bridge:** SquadOrchestrator é CommonJS, dashboard é ESM. Story 1.9 criou wrapper — pode precisar de ajustes para novos métodos. Mitigação: Estender wrapper existente.
- **EventStore access from dashboard:** EventStore roda no squad-engine (CommonJS). Dashboard precisa de bridge ou API endpoint para subscribir eventos. Mitigação: Criar API endpoint que squad-engine notifica, ou polling.
- **HMAC backward compatibility:** n8n existentes podem não enviar X-Webhook-Signature. Mitigação: Fallback para Bearer token quando signature ausente e `WEBHOOK_REQUIRE_SIGNATURE` não é true.
