## Story

**As a** squad-trafego (ou qualquer squad futuro),
**I want** chamar outro squad programaticamente, aguardar completion e receber outputs estruturados,
**so that** posso consumir criativos gerados por squad-copy automaticamente e orquestrar workflows multi-squad.

---

## Acceptance Criteria

1. **AC1 — Inter-Squad Trigger:** `SquadOrchestrator.executeSquad(squadId, options)` aceita `trigger.type = "inter_squad"` com `trigger.caller` identificando o squad chamador (ex: `"squad-trafego"`). Cria run com metadata de inter-squad relationship: `{ parentRunId, callerSquadId, callerStepId }`.

2. **AC2 — Wait For Completion:** `waitForCompletion(runId, options)` method que aguarda run completar usando polling do StateManager (interval: 5s, configurable). Timeout configurável (default: 30 minutos). Retorna `{ status, duration_ms, outputs_path }` on success. Throws `InterSquadTimeoutError` se timeout. Throws `InterSquadRunError` se run falha.

3. **AC3 — Get Outputs:** `getOutputs(runId)` retorna outputs estruturados do run: lê `outputs/` directory do run e retorna manifest com file paths. Formato: `{ runId, squadId, status, files: [{ path, type, size }], metadata: {} }`. Retorna `null` se run ainda não completou.

4. **AC4 — Inter-Squad Logging:** Logs mostram relationship entre runs. EventStore registra `inter_squad.call` (caller → callee), `inter_squad.completed`, `inter_squad.timeout`, `inter_squad.error`. Run details page mostra "Triggered by: squad-trafego (run xyz)".

5. **AC5 — Dashboard Inter-Squad Timeline:** Endpoint `GET /api/runs/{runId}/relationships` retorna parent/child run relationships. Dashboard run detail page mostra seção "Inter-Squad Calls" com caller info, callee runs, status, e duration.

6. **AC6 — Convenience Method:** `executeAndWait(squadId, options)` combina `executeSquad()` + `waitForCompletion()` + `getOutputs()` em uma chamada. Retorna `{ runId, status, duration_ms, outputs }`. Simplifica o caso de uso mais comum (chamar squad e esperar resultado).

7. **AC7 — Path Resolution Fix:** (Absorvido do backlog D-1740153600007 + D-1740153600008) SquadOrchestrator resolve paths de squad corretamente: busca em `squads/{squadId}/` (project root) e em `.aios-core/squads/{squadId}/` como fallback. Task files resolvidos relativamente ao squad directory. Fix necessário para inter-squad funcionar com qualquer squad.

### Inter-Squad Flow

```
Squad-Trafego (caller)                    Squad-Copy (callee)
  │                                          │
  ├─ executeAndWait("squad-copy", {          │
  │    trigger: {                            │
  │      type: "inter_squad",               │
  │      caller: "squad-trafego",           │
  │      parentRunId: "run-xyz"             │
  │    },                                    │
  │    timeout: 1800000  // 30 min          │
  │  })                                      │
  │     │                                    │
  │     ├──► executeSquad()                 ─┤──► Run created
  │     │    emit inter_squad.call           │    trigger.type = "inter_squad"
  │     │                                    │    trigger.caller = "squad-trafego"
  │     ├──► waitForCompletion()             │
  │     │    poll every 5s                   ├──► Steps executing...
  │     │    ...                             │    ...
  │     │    ...                             ├──► Run completed
  │     │    status = "completed" ◄──────────┤
  │     │                                    │
  │     ├──► getOutputs()                    │
  │     │    read outputs/ manifest          │
  │     │                                    │
  │     └──► return { runId, outputs }       │
  │                                          │
  ├─ use outputs.files[...]                  │
```

### Risks

- **Polling overhead:** waitForCompletion polls StateManager every 5s. Para runs longos (20-30min), são ~360 polls. Mitigação: Polling é lightweight (read JSON file). Futuramente pode ser substituído por event-based notification.
- **Deadlock potencial:** Squad A chama Squad B que chama Squad A. Mitigação: Para MVP, não implementar detection (documentar como known limitation). Phase 4 pode adicionar dependency graph.
- **Memory leak em timeout:** Se waitForCompletion timeout, o interval timer deve ser cleaned up. Mitigação: Usar try/finally com clearInterval.
