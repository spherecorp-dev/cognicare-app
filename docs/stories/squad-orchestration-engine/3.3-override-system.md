## Story

**As a** AIOS user (ou n8n workflow),
**I want** passar overrides ao triggerar um run para modificar comportamento padrão do pipeline,
**so that** posso forçar método específico, filtrar geos, selecionar plataformas, ou pular fases sem alterar o playbook.

---

## Acceptance Criteria

1. **AC1 — Override Field in Trigger:** Todos os trigger types (manual, webhook, inter_squad) aceitam campo `overrides: { method?, geos?, platforms?, skip_phases?, custom?: {} }`. Overrides são armazenados no run state em `trigger.overrides` e propagados para todos os steps via `context.overrides`.

2. **AC2 — Override Propagation to Tasks:** TaskExecutor injeta `context.overrides` em cada task execution. Tasks podem acessar overrides via `context.overrides.{field}`. Se override não presente, task usa seu default. Pattern: `const method = context.overrides?.method || await agentDecide()`.

3. **AC3 — Method Override:** `overrides.method` (string: "modelagem" | "variacao_de_winner" | "do-zero") força o método de copy sem agente decidir. Task `select-method` checa `context.overrides.method` — se presente, retorna direto. Se ausente, agente decide normalmente.

4. **AC4 — Geo and Platform Filters:** `overrides.geos` (string[]: ["fr", "es", "en"]) filtra geos no pipeline. `overrides.platforms` (string[]: ["meta", "tiktok"]) filtra plataformas. Tasks de spy-scraping e delivery respeitam esses filtros.

5. **AC5 — Skip Phases:** `overrides.skip_phases` (string[]: ["intelligence", "strategy"]) pula fases inteiras do pipeline. Steps em fases skipadas são marcados como `skipped` com reason `"phase_skipped_by_override"`. Pipeline continua com próxima fase não-skipada.

6. **AC6 — Override Validation:** Antes de iniciar run, valida overrides contra schema:
   - `method` deve ser um dos valores permitidos (ou undefined)
   - `geos` deve ser array de strings ISO 639-1 válidas (ou undefined)
   - `platforms` deve ser array de strings de plataformas suportadas (ou undefined)
   - `skip_phases` deve ser array de phase names existentes no playbook (ou undefined)
   - `custom` é free-form object (para extensibilidade futura)
   - Validação inválida retorna `400 { error: "Invalid overrides", details: [...] }` no webhook, ou throws `OverrideValidationError` no API direto.

7. **AC7 — Dashboard Override Visibility:** Endpoint `GET /api/runs/{runId}` inclui `trigger.overrides` na response. Dashboard run detail page mostra seção "Applied Overrides" com badges visuais (ex: "Method: variacao_de_winner", "Geos: FR", "Skipped: intelligence"). Steps skipados por override mostram reason no step detail.

### Override Schema

```javascript
const OVERRIDE_SCHEMA = {
  method: {
    type: 'string',
    enum: ['modelagem', 'variacao_de_winner', 'do-zero'],
    optional: true
  },
  geos: {
    type: 'array',
    items: { type: 'string', pattern: /^[a-z]{2}$/ },  // ISO 639-1
    optional: true
  },
  platforms: {
    type: 'array',
    items: { type: 'string', enum: ['meta', 'tiktok', 'google', 'snapchat'] },
    optional: true
  },
  skip_phases: {
    type: 'array',
    items: { type: 'string' },  // validated against playbook phases
    optional: true
  },
  custom: {
    type: 'object',
    optional: true
    // free-form, no validation
  }
};
```

### Override Flow

```
Trigger (manual/webhook/inter_squad)
  │
  ├─ overrides: { method: "variacao_de_winner", geos: ["fr"] }
  │
  ▼
Override Validation (override-validator.js)
  │ ✓ method is valid enum value
  │ ✓ geos are valid ISO 639-1
  │
  ▼
Store in run state (trigger.overrides)
  │
  ▼
TaskExecutor injects context.overrides
  │
  ├─ Step: select-method
  │   └─ context.overrides.method = "variacao_de_winner" → return directly
  │
  ├─ Step: spy-scrape
  │   └─ context.overrides.geos = ["fr"] → filter to FR only
  │
  ├─ Step: generate-images (phase: production)
  │   └─ not in skip_phases → execute normally
  │
  └─ Dashboard: shows "Method: variacao_de_winner", "Geos: FR"
```

### Risks

- **Playbook phase names coupling:** skip_phases valida contra nomes de fases do playbook. Se playbook renomear fases, overrides antigos quebram. Mitigação: Validação é at-trigger-time, não persistida. Documentar phase names como API contract.
- **Override scope creep:** Custom field é free-form, pode virar dumping ground. Mitigação: Documentar que custom é para extensibilidade futura, não para bypass de validação.
- **Missing override check em tasks:** Se um task não checa context.overrides, override é silenciosamente ignorado. Mitigação: Adicionar log warning se override field existe mas nenhum task o consumiu.

### Concerns (LOW Severity — Non-Blocking)

**Concern #1: AC3 method override test is structural, not behavioral**
- Test verifies `context.overrides.method` is accessible but does NOT verify that `select-method` task actually bypasses agent decision.
- **Risk:** LOW — `select-method` task doesn't exist yet; test validates plumbing, not end behavior.
- **Recommendation:** When `select-method` is implemented, add test asserting direct return when `context.overrides.method` is set.

**Concern #2: AC4 geo/platform filter test is assertion-only**
- Test asserts data structure on a plain JS object — no execution through TaskExecutor.
- **Risk:** LOW — spy-scraping and delivery tasks don't exist yet; test validates contract.
- **Recommendation:** When filter-consuming tasks are implemented, add integration tests.

**Concern #3: Dual validation logic (DRY)**
- Override validation exists in `override-validator.js` (CommonJS) and `webhook-validator.ts` (TypeScript) with mirrored but independent implementations.
- **Risk:** MEDIUM — If ALLOWED_METHODS or ALLOWED_PLATFORMS change, both files need updating.
- **Recommendation:** Consider extracting shared constants to a JSON file importable from both contexts, or add a test that asserts parity between the two implementations.

**Concern #4: Context propagation contract**
- `state.trigger.overrides` is stored at trigger time, but the propagation to `context.overrides` in the step execution loop is implicit (depends on the loop reading state and building context).
- **Risk:** LOW — Tests validate both ends (storage and consumption). The contract is clear.
- **Recommendation:** Document that the step execution loop MUST map `state.trigger.overrides` → `context.overrides`.
