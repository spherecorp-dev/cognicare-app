## Story

**As a** Squad Orchestration Engine,
**I want** estado ativo em Redis para pause/resume rápido,
**so that** posso suportar 100+ runs simultâneos sem latência de File System I/O.

---

## Acceptance Criteria

1. **AC1 — Redis State Store:** `StateManager` usa Redis como store primário para estado ativo de runs. Key format: `run:{runId}` com valor JSON serializado do state. Redis connection compartilhada com BullMQ (Story 4.1).

2. **AC2 — Dual Write Strategy:** `save()` escreve em Redis E File System simultaneamente (dual write). Redis é source-of-truth para runs ativos. File System mantido para backup e historical access.

3. **AC3 — Read Strategy (Cache-First):** `load()` lê de Redis primeiro (cache hit). Se Redis miss (key expired ou Redis unavailable), fallback para File System. Log de cache hits/misses para observabilidade.

4. **AC4 — TTL Management:** Runs ativos (status: running, paused, queued) têm TTL de 24h em Redis. Runs completados/abortados são removidos de Redis após 1h (state já persistido em File System). TTL refresh em cada `save()`.

5. **AC5 — Graceful Degradation:** Se Redis unavailable (connection error, timeout): `load()` faz fallback transparente para File System, `save()` escreve apenas em File System com warning log. Nenhuma operação falha por Redis outage — apenas performance degrada.

6. **AC6 — Performance Targets:** Pause/resume com Redis completa em <100ms (vs ~500-1000ms com File System). `save()` latency <50ms (Redis) + async File System write. `load()` latency <20ms (Redis hit).

7. **AC7 — Metrics and Observability:** Expor métricas: `redis_hits`, `redis_misses`, `redis_errors`, `redis_latency_avg`, `fs_fallback_count`. Dashboard endpoint `GET /api/metrics` inclui cache stats. Logs estruturados para cada cache operation.

### State Serialization

```javascript
// Redis stores JSON (not YAML) for performance
// File System continues using YAML for human readability

// Redis: fast JSON parse/stringify
await redis.set(`run:${runId}`, JSON.stringify(state), { EX: ttl });
const state = JSON.parse(await redis.get(`run:${runId}`));

// File System: YAML for backup/historical (existing behavior)
fs.writeFileSync(path, yaml.dump(state));
```

### Risks

- **Data consistency:** Dual write pode resultar em inconsistência se Redis write sucede mas FS falha (ou vice-versa). Mitigação: Redis é source-of-truth para runs ativos; FS é backup. Se divergem, Redis wins.
- **Serialization format mismatch:** Redis usa JSON, FS usa YAML. Mitigação: State object é o mesmo, apenas serialization format difere. Desserialização trata ambos.
- **Redis memory:** 100+ state objects em Redis podem consumir memória. Mitigação: TTL garante cleanup automático. State objects são ~5-20KB cada.
- **Checksum validation:** SHA256 checksum do Story 1.4 é calculado sobre YAML. Redis usa JSON. Mitigação: Recalcular checksum sobre JSON representation, ou manter checksum no state object.

### Completion Notes

- **RedisStateAdapter**: New standalone adapter with get/set/delete/exists/ping + metrics tracking. Uses ioredis (transitive dependency from BullMQ). Key format: `run:{runId}` with JSON serialization.
- **Dual Write**: StateManager.saveState() writes to Redis (sync) and File System (sync). If Redis fails, FS write continues with warning log. No operation ever fails due to Redis outage.
- **Cache-First Read**: StateManager.loadState() tries Redis first (cache hit). On miss/error, falls back to File System and populates Redis cache on success.
- **TTL Management**: Active runs (running/paused/queued) = 24h TTL. Completed runs (completed/aborted/failed) = 1h TTL. TTL refreshes on every save(). Redis auto-evicts expired keys.
- **Graceful Degradation**: Inline circuit breaker (CLOSED/OPEN/HALF_OPEN). Threshold: 3 failures, reset timeout: 30s. All operations degrade to File System only when circuit is open.
- **Cache Metrics**: hits/misses/errors/avgLatency/fsFallbackCount tracked in RedisStateAdapter. Dashboard endpoints: GET /api/metrics, GET /api/health.
- **Performance**: Mock Redis tests confirm <20ms load (cache hit), <50ms save, <100ms pause/resume cycle. Real Redis performance to be validated in production.
- **Backward Compatibility**: 30/30 original state-manager tests pass unchanged. StateManager without redisAdapter behaves exactly as before.
- **Test Summary**: 39 RedisStateAdapter unit tests + 38 StateManager-Redis integration tests + 30 original StateManager tests = 107 tests, 0 failures. Dashboard: 207/207 PASS.
