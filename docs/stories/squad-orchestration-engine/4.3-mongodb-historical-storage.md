## Story

**As a** AIOS administrator,
**I want** histórico de runs em MongoDB para queries ricas e analytics,
**so that** posso analisar performance de squads, gerar relatórios, e identificar top failing tasks.

---

## Acceptance Criteria

1. **AC1 — MongoDB Collection:** Collection `squad_runs` criada em MongoDB Atlas (ou local). Schema definido com campos: `runId`, `squadId`, `trigger`, `status`, `started_at`, `completed_at`, `duration_ms`, `phases_summary`, `outputs_summary`, `events_count`, `error` (se failed).

2. **AC2 — Auto-Archive on Completion:** Quando um run atinge status final (completed, failed, aborted), estado é automaticamente arquivado em MongoDB. Archive inclui summary (não o state completo — state completo permanece em File System).

3. **AC3 — Indexes:** Indexes criados para queries comuns:
   - `{ squadId: 1, started_at: -1 }` — listar runs por squad
   - `{ status: 1 }` — filtrar por status
   - `{ started_at: -1 }` — ordenação cronológica
   - `{ "error.step": 1 }` — top failing tasks

4. **AC4 — Analytics API:** Endpoint `GET /api/analytics` retorna métricas agregadas:
   - Average duration por squad (24h, 7d, 30d)
   - Success rate por período (24h, 7d, 30d)
   - Top 5 failing tasks (step + error count)
   - Total runs por status (completed, failed, aborted)
   - Throughput (runs/dia, runs/semana)

5. **AC5 — Dashboard Analytics Page:** Página `/dashboard/analytics` com visualizações:
   - Cards: avg duration, success rate, total runs, active runs
   - Chart: runs por dia (últimos 30 dias) — bar chart com status colors
   - Table: top failing tasks com count e last seen
   - Filter: período (24h, 7d, 30d), squad (all, squad-copy, etc.)

6. **AC6 — Data Retention:** Policy de retenção configurável:
   - Default: 90 dias (runs mais antigos são automaticamente removidos)
   - Env var `MONGO_RETENTION_DAYS` para override
   - Cleanup job roda diariamente (ou via TTL index)
   - Log de cleanup: quantidade removida, espaço liberado

7. **AC7 — Query Performance:** Queries de analytics completam em <500ms para datasets de até 10,000 runs. Aggregation pipelines otimizados com indexes. Resultados de analytics cacheados em Redis (TTL 5min) para reduzir load no MongoDB.

### Risks

- **MongoDB availability:** Se MongoDB unavailable, archive falha silenciosamente. Mitigação: Log warning, state persiste em FS. Retry archive quando MongoDB recupera (eventual consistency).
- **Query performance:** Aggregation pipelines em 10,000+ docs podem ser lentos sem indexes. Mitigação: Indexes definidos upfront, Redis cache para queries frequentes (TTL 5min).
- **Data consistency:** Archive é eventual (triggered on completion). Se server crash entre completion e archive, run fica só em FS. Mitigação: Startup job que verifica FS runs não-archived e re-archiva.
- **Schema evolution:** Se schema mudar em stories futuras, docs antigos podem ter campos faltando. Mitigação: Schema validation é advisory, queries usam `$ifNull` para default values.
